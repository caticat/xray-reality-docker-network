# 三、Dockerfile（最小稳定版）

```dockerfile
FROM alpine:3.19

RUN apk add --no-cache curl bash

# 安装 Xray
RUN curl -L https://github.com/XTLS/Xray-core/releases/latest/download/Xray-linux-64.zip -o xray.zip \
    && unzip xray.zip -d /usr/local/bin/ \
    && rm xray.zip

RUN chmod +x /usr/local/bin/xray

WORKDIR /etc/xray

CMD ["/usr/local/bin/xray", "-config", "/etc/xray/config.json"]
```

# 七、如何“参数化”（你重点想要的）

你提到：

> “通过参数传递配置”

👉 有两种工程方案：

---

## ✅ 方案 A：env + 启动脚本生成 config（推荐）

### 改造 Dockerfile：

```dockerfile
COPY entrypoint.sh /entrypoint.sh
RUN chmod +x /entrypoint.sh

ENTRYPOINT ["/entrypoint.sh"]
```

---

### entrypoint.sh

```bash
#!/bin/bash

cat > /etc/xray/config.json <<EOF
{
  "inbounds": [{
    "port": ${PORT},
    "protocol": "vless",
    "settings": {
      "clients": [{
        "id": "${UUID}",
        "flow": "xtls-rprx-vision"
      }],
      "decryption": "none"
    },
    "streamSettings": {
      "network": "tcp",
      "security": "reality",
      "realitySettings": {
        "dest": "${DEST}",
        "serverNames": ["${SNI}"],
        "privateKey": "${PRIVATE_KEY}",
        "shortIds": ["${SHORT_ID}"]
      }
    }
  }],
  "outbounds": [{ "protocol": "freedom" }]
}
EOF

exec /usr/local/bin/xray -config /etc/xray/config.json
```

---

### docker-compose.yml（带参数）

```yaml
services:
  xray:
    build: .
    ports:
      - "443:443"
    environment:
      - PORT=443
      - UUID=xxxx-xxxx
      - DEST=www.microsoft.com:443
      - SNI=www.microsoft.com
      - PRIVATE_KEY=xxxx
      - SHORT_ID=abcd1234
```

---

## ✅ 方案 B：完全外部 config（最简单）

👉 不做任何模板
👉 直接挂载 config.json

优点：

* 最简单
* 最稳定

缺点：

* 不够“动态”

# 八、密钥自动生成（进阶）

你可以在 entrypoint 里加：

```bash
if [ -z "$PRIVATE_KEY" ]; then
  KEYS=$(/usr/local/bin/xray x25519)
  echo "$KEYS"
fi
```

👉 实现：

* 每次启动自动生成 key
* 或首次生成后持久化

# ✅ 一、vc2-1c-1gb 能不能跑容器？

# 🧠 二、底层原理（为什么可以）

你要理解一点（关键）：

👉 Docker 不是虚拟机
👉 它是 **Linux namespace + cgroups**

所以：

* 不需要额外 OS
* 几乎没有虚拟化开销 ([Vultr Community][2])

👉 在 VPS 上跑 Docker，本质就是：

```text
进程隔离 + 资源限制
```

---

# ⚠️ 三、但 vc2-1c-1gb 有真实限制

这个才是重点（很多教程不会说）

## 资源上限

你的机器：

* 1 vCPU
* ~950MB 可用内存 ([VPSBenchmarks][3])

---

### 实际可用情况：

| 项目            | 占用         |
| ------------- | ---------- |
| OS（Ubuntu）    | ~150–250MB |
| Docker daemon | ~50–100MB  |
| 剩余给容器         | ❗ 600MB 左右 |

---

👉 结论：

> ✔ 能跑
> ❗ 但只能跑“轻量容器”

# 🚀 四、跑 Reality 容器 —— 能不能？

👉 结论：

> ✅ **完全没问题，而且非常适合**

## 为什么 Reality 很轻？

Reality（xray-core）特点：

* CPU 占用极低
* 内存占用很小（几十 MB 级别）
* 无数据库
* 无复杂 IO

👉 实际运行情况：

| 指标  | Reality  |
| --- | -------- |
| CPU | 很低       |
| 内存  | ~20–50MB |
| 网络  | 主消耗      |

---

👉 所以：

> 在 1GB 机器上几乎“感觉不到负担”

# ⚠️ 五、容器 vs 裸跑（重要建议）

你问的是容器，我给你一个**真实工程建议**：

---

## ✅ 可以用 Docker（优点）

* 部署快
* 配置可复用
* 更新方便
* 易迁移

---

## ❗ 但 Reality 场景其实不一定推荐容器

原因：

### 1️⃣ 网络层复杂化

Reality 涉及：

* TLS
* 指纹伪装
* TCP/UDP

👉 Docker 会引入：

* NAT
* bridge 网络

👉 可能带来：

* 延迟增加
* MTU 问题
* debug 更麻烦

---

### 2️⃣ 你是单服务

👉 没必要容器化

---

## 👉 工程结论：

| 方式               | 推荐    |
| ---------------- | ----- |
| 直接跑 xray         | ✅ 最优  |
| Docker 跑 reality | ✅ 可以  |
| 多容器编排            | ❌ 没必要 |
